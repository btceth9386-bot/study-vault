# Highlights

- `Section: Performance vs Scalability` Performance problem: system is slow for a single user. Scalability problem: system is fast for a single user but slow under heavy load. A service is scalable if adding resources results in proportional performance increase.
- `Section: CAP Theorem` Networks are inherently unreliable, so partition tolerance must be supported. This forces a trade-off between consistency and availability. CP systems wait for partition resolution (banks, inventory); AP systems serve stale data (social media, DNS).
- `Section: Consistency Patterns` Weak consistency (memcached, VoIP): reads may not see recent writes. Eventual consistency (DNS, Cassandra): reads eventually see writes within milliseconds. Strong consistency (RDBMS, ZooKeeper): reads immediately see writes via synchronous replication.
- `Section: Availability Patterns` Active-passive failover: heartbeat monitoring, passive takes over IP on failure. Active-active: both servers handle traffic, no switchover downtime. Series availability decreases (A×B), parallel availability increases (1-(1-A)(1-B)).
- `Section: Load Balancer` Layer 4 LB: routes by IP+port via NAT, no packet inspection. Layer 7 LB: terminates connection, inspects HTTP headers/content, enables content-based routing (video → video servers, billing → secure servers).
- `Section: Database Scaling` Master-slave: master serves writes, slaves serve reads; potential data loss on master failure. Sharding: distribute data subsets across databases; benefits include less traffic, more cache hits, smaller indexes; challenges include routing complexity and cross-shard joins.
- `Section: Caching Strategies` Cache-aside: app reads cache → miss → query DB → update cache (3 round trips on miss). Write-through: app writes to cache, cache synchronously writes to DB (slower writes, fast reads). Write-behind: async DB writes from cache (risk of data loss). Refresh-ahead: proactive refresh before TTL expiry (worse if predictions inaccurate).
- `Section: Async Processing` Message queues (Redis, RabbitMQ, SQS): publish job → immediate response → worker processes async. Back pressure: limit queue size, return HTTP 503 when full, clients use exponential backoff. Prevents memory exhaustion and cascading failures.
- `Section: RPC vs REST` RPC is action-oriented (POST /signup), tightly coupled, optimized for internal services. REST is resource-oriented (POST /persons), loosely coupled, stateless, cacheable, ideal for public APIs.
- `Section: Twitter Design` Fanout service is the bottleneck: celebrity tweets take minutes to fan out. Solution: hybrid approach — pre-compute timelines for most users, compute on-read for high-follower accounts. Redis timeline structure: 8 bytes tweet_id + 8 bytes user_id + 1 byte meta per entry.
- `Section: Pastebin Design` URL generation: MD5 hash of IP+timestamp → Base62 encode → take first 7 chars (62^7 = 3.5 trillion possible values). Polyglot persistence: SQL for metadata (shortlink, created_at, paste_path), S3 for paste content.
- `Section: AWS Scaling` Progressive scaling stages: single server → vertical scaling → horizontal + LB → DB replication → caching (ElastiCache) → CDN (CloudFront) → stateless architecture → sharding → async (SQS) → multi-region. Each stage addresses a specific bottleneck.
- `Section: SQL vs NoSQL` Choose SQL when: structured data, strict schema, complex joins, ACID transactions. Choose NoSQL when: semi-structured data, flexible schema, no complex joins, TB/PB scale, high IOPS. BASE (Basically Available, Soft state, Eventual consistency) vs ACID.
