# Highlights

- `Introduction.md` DSPy's core thesis: "Prompts are implementation details." Instead of writing prompt strings, declare task structure with Signatures, compose Modules, define a metric, and let an optimizer compile the program — the same workflow as training an ML model.
- `Signatures__Task_Definition.md` String shorthand: `dspy.Signature("question -> answer")` vs class form with typed `InputField`/`OutputField` and a docstring that becomes the task instruction. Signatures are immutable; `with_instructions()`, `prepend()`, `append()` return new classes — enabling programmatic mutation without side effects.
- `Predict_Module.md` `dspy.Predict` stores two things: the **instruction** (from Signature docstring, mutable by optimizers) and **demonstrations** (few-shot examples, populated by optimizers). Everything else — formatting, adapters, LLM calls — is derived from these two parameters. This is why state serialization is so clean.
- `Module_System__Base_Classes.md` `ProgramMeta` metaclass ensures `_base_init()` always runs before user `__init__`, populating bookkeeping fields. `named_predictors()` recurses through the module tree to discover all optimizable predictors — this is how a single `optimizer.compile(program)` call can tune all nested predictors simultaneously.
- `Building_DSPy_Programs.md` The compile-then-run lifecycle: (1) write program as Python class, (2) define metric as `(example, prediction, trace) -> float`, (3) call `optimizer.compile(program, trainset=..., metric=...)`, (4) save with `program.save("optimized.json")`, (5) load in production. The compiled artifact is model-agnostic — it can be recompiled for a different LLM.
- `Few-Shot_Optimizers.md` `BootstrapFewShot` execution flow: run teacher program on `trainset` → capture traces of successful calls → filter by `metric_threshold` → assign best traces as demonstrations to matching predictors in the student. Temperature is raised on retry rounds via a fresh `rollout_id` to bypass cache and get diverse completions.
- `Few-Shot_Optimizers.md` `BootstrapFewShotWithRandomSearch` generates three fixed baselines (zero-shot, labeled-only, unshuffled bootstrap) plus `num_candidate_programs` additional runs with shuffled trainsets, evaluates all on a validation set, and returns the best. This makes optimization into a search problem over the demo configuration space.
- `GEPA__SIMBA_Reflective_and_Stochastic_Optimization.md` `SIMBA` does stochastic mini-batch ascent: for each batch, it identifies examples with high performance variability, then uses two strategies to improve: `append_a_demo` (add a successful trace as a demonstration) or `append_a_rule` (compare good vs bad traces, ask an LLM to articulate a textual improvement rule and append it to the instruction). Rules accumulate over steps.
- `Fine-tuning__Weight_Optimization.md` `BootstrapFinetune` workflow: compile a teacher program (e.g., GPT-4o-based) with `BootstrapFewShot` → execute on a large trainset to gather (input, output) pairs → fine-tune a target model (e.g., Llama-3.2-1B) on those pairs via a provider (`LocalProvider`, `DatabricksProvider`, `OpenAIProvider`) → return a new DSPy program using the fine-tuned model. Shopify achieved 550× cost reduction this way.
- `Adapter_System.md` Adapters bridge Signatures and LLM prompts. `ChatAdapter` (default) uses `[[ ## field_name ## ]]` delimiters; `JSONAdapter` enforces JSON schema with structured outputs when the provider supports it; `TwoStepAdapter` handles reasoning models (o1/o3) by separating the reasoning step from field extraction. Adapters handle both formatting input and parsing output back into typed fields.
- `Assertions__Output_Validation.md` Two validation primitives: `dspy.Assert(condition, msg)` is fatal — raises `AssertionError` on violation. `dspy.Suggest(condition, msg)` is soft — triggers a backtrack/retry (up to `max_backtracks` times) with the failure message injected into the next attempt as feedback. `dspy.Retry` wraps any module for retry-with-feedback without modifying the module's code.
- `Caching__Performance_Optimization.md` Two-tier cache: in-memory LRU (O(1), per-process) → disk `FanoutCache` (persistent across restarts). Cache key = SHA-256 of request dict minus ignored fields (`api_key`, `api_base`). Use `rollout_id` in `dspy.settings` to bypass the cache for stochastic optimization runs without disabling caching globally.
- `Evaluation_Framework.md` `dspy.Evaluate(devset=..., metric=..., num_threads=8)` runs the module on a dataset in parallel and aggregates scores. Built-in metrics: exact match (`EM`), `F1`, `HotPotF1`. LLM-as-judge metrics: `SemanticF1`, `CompleteAndGrounded`. Evaluation results include per-example `(example, prediction, score)` tuples for debugging failures.
- `Use_Cases__Applications.md` Four canonical use cases: (1) multi-hop reasoning — chain retrieval + synthesis steps; (2) tool-augmented agents — ReAct loop with function calling; (3) structured extraction with typed `Literal` outputs; (4) classification + distillation — compile with large model, fine-tune small model for production cost.
- `Parallel__Async_Execution.md` `dspy.Parallel(num_threads=8)` runs a list of `(module, example)` pairs concurrently using `ThreadPoolExecutor`, propagating `dspy.settings` context to all threads. `asyncify(module)` wraps a sync module into an async one using `anyio.CapacityLimiter` to avoid saturating the event loop. `streamify(module)` yields `StatusMessage`, `StreamResponse`, and finally `Prediction` tokens via async memory channels.
